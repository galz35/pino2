2.4 MODIFICAR: orders.service.ts
==================================
Cambios necesarios en el servicio existente:

a) create() → Lógica nueva:
   - Recibir campo tipo_pedido (VENTA_ESTANDAR | ABASTECIMIENTO_INTERNO | ENTREGA_POR_CUENTA)
   - Si tipo_pedido = ABASTECIMIENTO_INTERNO:
     * No generar comisión
     * Usar precio de costo
     * Marcar con prioridad alta
   - Si tipo_pedido = ENTREGA_POR_CUENTA:
     * requiere_cobro = false
     * No genera cuenta por cobrar
     * El ticket sale en C$ 0.00 (remisión)
   - Si priceLevel >= 4:
     * requiere_autorizacion = true
     * status = 'PENDIENTE_AUTORIZACION' (no 'RECIBIDO')
   - VALIDACIÓN DE MORA CRUZADA:
     * Consultar grupo_economico_id del cliente
     * Si cualquier cliente del grupo tiene factura vencida → HTTP 403
     * Mensaje: "Mora activa en sucursal hermana: [nombre del local en mora]"
   - CONCURRENCIA OPTIMISTA:
     * SELECT * FROM products WHERE id = X FOR UPDATE (ya parcialmente implementado)

b) NUEVO: updateStatus() → Agregar transición PENDIENTE_AUTORIZACION:
   Mapa de transiciones actualizado:
     PENDIENTE_AUTORIZACION → RECIBIDO (cuando se autoriza) | CANCELADO (cuando se rechaza)
     RECIBIDO → EN_PREPARACION | CANCELADO
     EN_PREPARACION → ALISTADO | CANCELADO
     ALISTADO → CARGADO_CAMION
     CARGADO_CAMION → EN_ENTREGA
     EN_ENTREGA → ENTREGADO | RECHAZO_TOTAL | DEVUELTO
     ENTREGADO → LIQUIDADO
     RECHAZO_TOTAL → LIQUIDADO

c) NUEVO: autorizar(orderId, userId, decision: 'aprobar' | 'rechazar')
   - Si aprobar: status → RECIBIDO, guardar autorizado_por y fecha_autorizacion
   - Si rechazar: status → CANCELADO
   - Notificar al preventa vía push

2.5 MÓDULO: cargas-camion (NUEVO)
===================================
Archivo: backend/src/modules/cargas-camion/

Endpoints:
  POST   /api/cargas-camion                → Crear carga (asignar rutero + camión)
  GET    /api/cargas-camion?storeId=&fecha= → Listar cargas del día
  GET    /api/cargas-camion/:id            → Detalle con pedidos y consolidación
  PUT    /api/cargas-camion/:id/salida     → Marcar salida (EN_RUTA)

Lógica clave en GET /:id (Consolidación Matemática):
  - Agrupar TODOS los pedidos de la carga
  - Para cada producto:
    * cantidad_total = SUM(quantity) de todos los pedidos
    * bultos = FLOOR(cantidad_total / unidades_por_bulto)
    * unidades_sueltas = cantidad_total % unidades_por_bulto
  - Retornar DOS arrays:
    * lista_bultos: [{ producto, bultos, unidadesPorBulto }]
    * lista_unidades: [{ producto, unidadesSueltas }]

2.6 MÓDULO: asignacion-masiva (NUEVO)
========================================
Endpoint POST /api/despacho/asignar-masivo

Body: {
  storeId: string,
  ruteroId: string,
  camionPlaca: string,
  orderIds: string[],     // Array de IDs de pedidos
  fechaEntrega: string
}

Lógica:
  1. Crear registro en cargas_camion
  2. Para cada orderId: UPDATE orders SET rutero_id, camion_id, grupo_carga_id, fecha_entrega_programada
  3. Cambiar status de todos a LISTO_PARA_ALISTAR (o ALISTADO si ya están alistados)
  4. Emitir evento WebSocket CARGA_ASIGNADA
  5. Notificar al rutero asignado

2.8 MÓDULO: liquidaciones (NUEVO)
====================================
Archivo: backend/src/modules/liquidaciones/

Endpoints:
  POST /api/liquidaciones                → Crear liquidación de ruta
  GET  /api/liquidaciones?storeId=&fecha= → Listar
  GET  /api/liquidaciones/:id            → Detalle con desglose

Lógica en POST:
  - Recibe rutero_id, fecha
  - Calcula automáticamente:
    * total_pedidos = COUNT de pedidos asignados al rutero ese día
    * total_entregados = COUNT con status ENTREGADO
    * total_rechazados = COUNT con status RECHAZO_TOTAL
    * total_cobrado_contado = SUM de pagos contado del día
    * total_cobrado_credito = SUM de cobros de cuentas a crédito
    * efectivo_esperado = contado + cobros_credito - devoluciones
  - Compara contra el arqueo (si existe)
  - Calcula diferencia

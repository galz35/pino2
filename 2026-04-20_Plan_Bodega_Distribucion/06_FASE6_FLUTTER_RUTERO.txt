CONCEPTO CLAVE:
  El Rutero recibe el camión cargado y sale a entregar.
  Cobra CONTADO al momento de entregar.
  Cobra CRÉDITO 8 días después de la entrega.
  Puede recibir rechazos totales o parciales.
  Al final del día hace cierre/liquidación de ruta.

6.2 NUEVO: delivery_detail_screen.dart
=========================================
Archivo: flutter/lib/features/deliveries/presentation/screens/delivery_detail_screen.dart

Flujo de entrega completo:
  1. Rutero toca un pedido pendiente
  2. Se muestra detalle:
     ┌──────────────────────────────────┐
     │ 📦 Pedido #abc123                │
     │ Cliente: Pulpería Doña María     │
     │ 📍 Bo. El Centro, Calle Principal│
     │ [🗺 Ver en Mapa]                 │
     ├──────────────────────────────────┤
     │ ITEMS A ENTREGAR:               │
     │ ✅ Coca Cola 600ml  x24   C$120 │
     │ ✅ Aceite 1L        x2    C$ 50 │
     │ ✅ Galleta María    x12   C$ 60 │
     ├──────────────────────────────────┤
     │ Total: C$ 230.00                │
     │ Tipo: CONTADO                    │
     ├──────────────────────────────────┤
     │ [🟢 ENTREGAR Y COBRAR]          │
     │ [🔴 RECHAZO TOTAL]              │
     └──────────────────────────────────┘

  3. Si ENTREGAR Y COBRAR (tipo VENTA_ESTANDAR + CONTADO):
     → Modal de cobro:
       - Monto a cobrar: C$ 230.00
       - Efectivo recibido: [_______]
       - Cambio: C$ XX.XX (calculado)
       - Botón "Confirmar Cobro"
     → PATCH /api/orders/:id/status { status: 'ENTREGADO' }
     → POST /api/collections { orderId, amount, paymentMethod: 'EFECTIVO' }

  4. Si ENTREGAR (tipo VENTA_ESTANDAR + CRÉDITO):
     → Confirmar entrega SIN cobro
     → Se genera automáticamente cuenta por cobrar (ya implementado en backend)
     → Mostrar: "Cobro programado para [fecha + 8 días]"
     → PATCH /api/orders/:id/status { status: 'ENTREGADO' }

  5. Si ENTREGAR (tipo ABASTECIMIENTO_INTERNO):
     → Solo confirmar entrega
     → Sin cobro, sin cuenta por cobrar
     → PATCH /api/orders/:id/status { status: 'ENTREGADO' }

  6. Si ENTREGAR (tipo ENTREGA_POR_CUENTA):
     → Solo confirmar entrega
     → Ticket de remisión (C$ 0.00)
     → Flag requiere_cobro = false
     → PATCH /api/orders/:id/status { status: 'ENTREGADO' }

  7. Si RECHAZO TOTAL:
     → Campo obligatorio: Motivo del rechazo
     → Opciones: "Cliente ausente", "No tiene dinero", "Producto dañado", "Otro"
     → PATCH /api/orders/:id/status { status: 'RECHAZO_TOTAL' }
     → Los items vuelven al inventario del rutero (para devolver a bodega)

6.5 NUEVO: devoluciones_screen.dart
======================================
Archivo: flutter/lib/features/returns/presentation/screens/route_returns_screen.dart

Propósito: Gestionar los items rechazados que el rutero trae de vuelta.

Flujo:
  1. Al rechazar un pedido, los items se agregan a una "lista de devoluciones"
  2. Al regresar a bodega, el rutero presenta esta lista
  3. El bodeguero confirma la recepción de cada item
  4. Se reintegran al inventario de bodega (products.current_stock += qty)
  5. Se descuentan del vendor_inventory del rutero

6.6 MEJORAS UX PARA RUTERO
==============================
  - Navegación GPS: Botón para abrir ruta optimizada en Google Maps/Waze
  - Firma digital del cliente al momento de la entrega (canvas de firma)
  - Foto de evidencia de entrega (opcional)
  - Modo offline robusto:
    * Todas las entregas y cobros se guardan localmente
    * Al recuperar señal, se sincronizan en orden cronológico
    * Conflictos se resuelven con timestamp del servidor
  - Notificación push cuando le asignan nueva ruta
  - Alerta sonora cuando hay cobros de crédito vencidos
